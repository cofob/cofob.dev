# cofob.dev contributor and agent guide

These instructions apply to the entire repository. Keep them current when blog authoring, media processing, feed rendering, or deployment behavior changes.

## Project shape

- This is a SvelteKit 2 / Svelte 5 personal site. Use Node.js 24.11 or newer.
- Blog posts are MDsveX files in `src/lib/blog/posts/`.
- Blog metadata and components are exposed through the virtual modules generated by `scripts/blog-content-plugin.js`.
- Blog media preparation is implemented in `scripts/blog-assets.js`; do not duplicate that processing in components.
- RSS and Atom contain complete post bodies. They render post components in `portable` mode through `src/lib/blog/render-mode.ts`.
- The supported deployment targets are Node, static/IPFS, and Cloudflare. Changes must not assume that a runtime server is always available.

Do not edit generated output in `.blog-build/`, `.svelte-kit/`, `build/`, or `node_modules/`. Change the source or preparation script and regenerate it. Preserve unrelated working-tree changes.

## Development and verification

Install dependencies and start the development server with:

```sh
npm ci
npm run dev
```

The development command prepares local assets, includes drafts and scheduled posts, and watches post and media changes.

Use the smallest relevant checks while developing, then run the broader checks appropriate to the change:

```sh
npm run check
npm run lint
npm run test:unit
npm run test:a11y
npm run test:smoke
npm run test:static
npm run test:cloudflare
```

`npm test` runs the main check, lint, unit, Chromium accessibility, Node build, and smoke-test sequence. Install Chromium once with `npx playwright install chromium`. Asset or routing changes should also run the static test; deployment-specific changes should run the Cloudflare test.

Local asset mode is the default and is safe for routine development. Do not set `BLOG_ASSET_MODE=external` unless the task explicitly requires uploading generated assets to the configured S3/R2 bucket. External objects are content-addressed and retained rather than automatically deleted.

## Creating a blog post

1. Copy `src/lib/blog/posts/example-post.md` to `src/lib/blog/posts/<slug>.md`.
2. Use a lowercase kebab-case filename. The filename is the public slug.
3. Keep the frontmatter schema exact; unknown fields fail validation.
4. Put post-specific local assets in `static/blog/<slug>/`.
5. Write the body in Markdown/MDsveX. Start body headings at `##` because the article layout supplies the `h1`.
6. Preview with `npm run dev`, initially keeping `draft: true`.
7. Run checks and set `draft: false` only when the post is ready. A future `published` time remains scheduled until a build occurs after that time.

Frontmatter fields:

- `title`: required, non-empty display title.
- `description`: required, non-empty summary used by cards, metadata, and feeds.
- `published`: required timezone-qualified ISO date-time.
- `updated`: optional timezone-qualified ISO date-time; it cannot precede `published`.
- `lang`: two- or three-letter language code with optional subtags; defaults to `en`.
- `tags`: up to 12 non-empty labels, each at most 40 characters and unique case-insensitively.
- `draft`: required boolean.
- `cover` and `coverAlt`: optional pair; either supply both or neither.
- `socialImage` and `socialImageAlt`: optional pair; either supply both or neither.
- `comments`: optional HTTPS Mastodon status, Pleroma/Akkoma `/notice/`, or `/users/.../statuses/...` URL. Do not import the comments UI into a post manually.

Keep prose as Markdown where possible. Put imports and data needed by MDsveX components in a top-level `<script>` block. Do not put an `h1` in the post body.

## Post media and exact optimization behavior

### Post-specific images and attachments

- Store local files under `static/blog/<slug>/` and reference them with paths relative to the post, for example `![Build succeeded](build.png)`.
- Local post and sticker references are not allowed to escape `static/blog/<slug>/` or `static/stickers/`.
- Displayable raster formats are JPEG, PNG, WebP, AVIF, and GIF. The build auto-orients them and emits WebP at quality `82`, effort `4`.
- Responsive widths are exactly `480`, `960`, and `1440` pixels when smaller than the source. The source width is added as the largest candidate when it is below 1440; images are never upscaled by the build.
- Animated GIF/WebP input is opened in animated mode and processed through the same responsive pipeline.
- SVG display images remain SVG and retain their measured intrinsic dimensions.
- Generated filenames include the first 12 hexadecimal characters of a SHA-256 content hash. External artifacts use `public, max-age=31536000, immutable`.
- A displayed raster source is not published in its original format unless the post also links to it as a download. Linked PDFs, audio, video, archives, and other attachments are copied unchanged with a content-hashed name.
- Never commit generated responsive variants. Commit only the source asset and let `npm run prepare:blog`, `npm run dev`, or a build create variants.

Markdown images must have meaningful alt text. The build rejects `![](...)`. For a genuinely decorative image, use explicit HTML with `alt=""`; do not evade the check with meaningless text.

### Covers and social images

- A cover uses the ordinary responsive-image pipeline and needs `coverAlt`.
- If no `socialImage` is provided, the build generates a 1200×630 PNG card.
- A local `socialImage` is auto-oriented, cropped with attention positioning, resized to exactly 1200×630, and emitted as a PNG.
- A remote social override must be an absolute HTTPS URL. It is treated as an already-prepared 1200×630 asset and is not downloaded or transformed.

### Stickers

Reusable stickers belong in `static/stickers/<pack-slug>/<sticker-name>.webp`; never copy the same pack asset into individual post folders.

The canonical committed sticker format is:

- exactly `192×192` pixels;
- square WebP, preferably with a transparent background;
- lowercase descriptive filename;
- content fitted without stretching or cutting off meaningful artwork.

When normalizing a new source, auto-orient it, fit it inside a transparent 192×192 canvas, and encode WebP with quality `82` and effort `4`. If the source is smaller than 192×192, find a better source rather than enlarging a visibly blurry sticker. The build then emits one content-hashed `192w.webp` display asset. `Sticker.svelte` declares 192×192 intrinsic dimensions and displays it at 128×128 on small screens and 160×160 from the `sm` breakpoint.

Every sticker use must include:

- `src`: absolute `/stickers/<pack>/<file>.webp` source path;
- `alt`: what the sticker conveys in this context, not merely “sticker”;
- `sourceName`: human-readable pack or artist name;
- `sourceUrl`: canonical HTTPS attribution URL.

Example:

```svelte
<Sticker
	src="/stickers/example-pack/reaction.webp"
	alt="Character reacting with delighted surprise"
	sourceName="Example Pack"
	sourceUrl="https://example.com/example-pack"
/>
```

Do not omit attribution, and do not bake attribution text into the bitmap.

## Common blog components

Import reusable article components from `src/lib/components/blog/` rather than recreating their visuals in a post.

- `WarningBlock`: prominent warning callout. Its optional `portableTitle` changes only the RSS/Atom heading.
- `NoticeBlock`: informational callout. Its optional `portableTitle` changes only the RSS/Atom heading.
- `Sticker`: optimized shared sticker with required alt text and source attribution.
- `ChatThread`: accepts `author`, `avatar`, `messages`, and an accessible `label`. Each message may have `text`, `link`, and `linkLabel`. Feed output becomes an avatar-free blockquote.
- `AsciinemaPlayer`: accepts `src`, optional `cols`/`rows`, and a descriptive `label`. `src` must be an absolute HTTPS URL on exactly `site-assets.cofob.dev`; feeds link to `/blog/play_asciinema/?url=...`.
- `ResponsiveImage`: use for image metadata supplied by the catalog or asset manifest. Ordinary post images should use Markdown so the build can rewrite them automatically.
- `Comments`, `CommentThread`, `CommentMedia`, and `RichText`: page-level Fediverse comment infrastructure, not components to embed in post Markdown.
- `PostCard` and `LatestPostLink`: blog listing/home-page components, not article-body components.

Use descriptive component `label` values in the language of the post. Prefer data arrays in the post’s script block over repeating large component markup.

## Adding or changing a blog component

Normal browser behavior and portable feed behavior are separate contracts:

1. The default mode is `browser`. Preserve existing visuals, content, links, loading behavior, and interaction unless the task explicitly changes them.
2. Read the mode with `getBlogRenderMode()` from `$lib/blog/render-mode` during component initialization.
3. If the browser component is complex or interactive, add a `portable` branch using basic SSR-safe HTML: headings, paragraphs, blockquotes, lists, links, images, figures, tables, `pre`, and `code`.
4. Portable output must contain the meaningful content, work without CSS or JavaScript, omit decorative duplication such as chat avatars, and replace unavailable interaction with a useful link.
5. Do not fetch remote data, access `window`, or rely on `onMount` to create portable content. RSS and Atom render on the server.
6. Preserve link labels and line breaks. Include alt text, captions, warnings, and source attribution.
7. Add browser-regression assertions and portable-render assertions to `src/lib/blog/post-render.test.ts`.
8. If the component appears in a published post, add complete RSS and Atom assertions to `src/lib/blog/feed.test.ts`.

Feed rendering passes `createPortableBlogContext()` to Svelte’s server renderer. Do not detect feeds through user agents, environment globals, or URL checks.

## Accessibility requirements

- Use semantic HTML first. Maintain one page `h1`; post content begins at `h2` and does not skip heading levels without a structural reason.
- Give every meaningful image concise contextual alt text. Use empty alt only for decoration. Do not repeat nearby captions verbatim in alt text.
- Give figures captions where the source or interpretation matters. Give terminal recordings and chat excerpts descriptive labels.
- Links must make sense out of context. Avoid “click here” and bare URLs when a useful label is available.
- Use real buttons for actions and real links for navigation. All interaction must work by keyboard and expose visible focus.
- Do not convey meaning by color alone. Keep text/background contrast at WCAG AA or better.
- Respect `prefers-reduced-motion`; the global stylesheet suppresses animation, and new motion must remain understandable when removed.
- Tables need header cells and should have a caption or nearby explanation. MDsveX automatically makes overflowing tables and code blocks keyboard-focusable regions; do not remove that behavior.
- Video and audio need controls, a descriptive label or caption, and usable fallback link text. Provide captions/transcripts when speech or sound carries information.
- New-window links must use `rel="noopener noreferrer"`; remote user-generated links should also use `nofollow` and an appropriate `referrerpolicy`.
- Do not inject untrusted HTML. Fediverse rich text must continue through the existing sanitized node model rather than `{@html}`.
- Test representative desktop and mobile widths with the existing axe/Playwright suite.

## RSS and Atom invariants

- `/rss.xml` and `/atom.xml` contain every public post and complete portable article HTML.
- A post's social image must be the first image in its portable article HTML. RSS also exposes it through Media RSS metadata; Atom exposes it as a `rel="enclosure"` link. Do not let an inline image or sticker become the inferred feed cover.
- RSS uses `content:encoded` CDATA; Atom uses escaped `<content type="html">`. Preserve the existing XML escaping and RSS `]]>` splitting.
- Every URL inside feed article HTML must be absolute. Keep URL normalization in `absolutizeHtmlUrls`, including every `srcset` candidate.
- RSS and Atom titles, canonical links, IDs/GUIDs, publication/update timestamps, language, categories, descriptions, and self-links must remain present.
- Do not include interactive comment threads in feeds.
- Feed HTML must not contain scripts, loading-only placeholders, browser player DOM, chat avatars, or other content that depends on site CSS/JavaScript.
- Validate both formats with `htmlparser2.parseFeed`, assert item count against `getPublicPosts()`, and test the actual endpoint content types and production/static artifacts.

## Svelte and styling conventions

- Use Svelte 5 runes (`$props`, `$state`, `$derived`) and TypeScript in components.
- Components used in posts must server-render without touching browser globals during initialization. Put browser-only work in `onMount`.
- Use `$app/paths.resolve` for typed internal navigation in Svelte pages/components. Feed-only URLs are made absolute later by the feed renderer.
- Component PostCSS should begin with `@reference "../../app.css"` (adjusted for depth) before using Tailwind utilities.
- Keep responsive layouts usable at narrow widths and avoid fixed content widths that cause page-level horizontal scrolling.
- External links opened in a new tab require the appropriate `rel` values.

When a rule here and the implementation disagree, treat that as a bug: verify the intended behavior, update the implementation or this guide, and add a regression test.
